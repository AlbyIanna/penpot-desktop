> **2026-07-15 fixpoint fix.** The materialization was NOT a round-trip fixpoint
> for the two asset-heavy v3 templates (tokens-starter-kit, penpot-design-system)
> because the settle was format-gated (`needs_settle()` fired only for legacy).

The first foothold of the package ecosystem (`docs/ecosystem-concept.md`): an
OFFLINE template gallery (the "Arduino Examples menu" move) seeded from the 15
builtin-template binfiles the runtime bundle already ships at
`runtime/backend/builtin-templates/`, and a "New file from template" verb that
imports one as a REAL working `.penpot` file into the active vault's default
project. "Surface, don't apply": nothing is applied silently — the user
explicitly creates a new file, and the result is a real folder on disk (folder
= source of truth).

## Which templates ship (and why none were dropped)

All **15** builtin templates ship. The N6 spike settled importability against
Penpot 2.16.2: the binfiles come in **two on-disk formats**, classified by
magic bytes, and BOTH import-as-new cleanly:

| Format | Magic | Count | Templates |
|---|---|---|---|
| binfile-v3 zip | `PK\x03\x04` | 4 | black-white-mobile-templates, penpot-design-system, plants-app, tokens-starter-kit |
| legacy binfile-v1 | `0x010B1A86` | 11 | avataaars, flex-layout-playground, font-awesome, lucide-icons, open-color-scheme, prototype-examples, tutorial-for-beginners, ux-notes, welcome, whiteboarding-kit, wireframing-kit |

PLAN2's floor was "4 minimum"; the whole gallery clears the bar, so nothing is
dropped. The catalog enumeration still classifies by magic at runtime and
drops (with a logged reason) anything that is neither format — the mechanism
exists, it just never fires for the shipped set.

Heavy-template note (carried from the spike, not re-litigated): the gallery
imports **lazily on click**, never preloads. `penpot-design-system` is 37 MB /
41 pages / 11338 boards — it imports fine but is slow and may approach the
multipart size limit in packaged mode (the chunked upload-session variant,
`docs/m0/rpc-endpoints.md:340`, is the documented fallback if it ever trips).
The new settle-until-fixpoint loop adds re-import/re-export cycles for it (it is
GC-dirty on first re-import, so it settles in 2 cycles); a full new-from-template
on it takes minutes — see the cost note under "Reported numbers".

## Format recipe (verified — the two penpot-rpc prerequisites)

Two changes in `crates/penpot-rpc` (previously exercised only on v3):

1. **`import_binfile_versioned`** (`src/lib.rs`) — sends an optional `version`
   multipart text field. `import_binfile` delegates with `None` (auto-detect,
   correct for v3). Legacy binfile-v1 REQUIRES `version=1` (omitting it → HTTP
   500; `version=3` → wrong-magic error). Unit + wire tests cover both.
2. **`sse::decode_import_end`** (`src/sse.rs`) — legacy imports answer with the
   transit-**set** shape `{"~#set":["~u<id>"]}` instead of the v3 array form
   `["~u<id>"]`. Both are now decoded. Unit tests cover the array, the set, and
   the rejection of an unexpected shape.

## New-from-template mechanism

`POST /__api/templates/new {templateId, name?}` (`apps/desktop/src/templates.rs`):

1. Validate `templateId` against the enumerated catalog (an EXACT entry — never
   a client-supplied path, so no traversal).
2. Resolve the active vault's default ("Drafts") project of the single team.
3. `import-binfile` as-new (no `file-id` → mints a fresh id, remaps internal
   ids), `version=1` for legacy origins. **Format only affects this `version`
   field** — it is NOT a reliable predictor of whether the first export is a
   fixpoint (see below), so it must not gate the settle.
4. **Settle until fixpoint (`settle_to_fixpoint`):** because import-as-new remaps
   ids there is NO hash equality with the source template, and the daemon
   materializes the tree by exporting the DB — so that FIRST on-disk export must
   itself be a round-trip fixpoint (export → normalize → in-place re-import →
   re-export → identical semantic hash per `scripts/roundtrip.py`), or the core
   invariant (delete DB, rebuild from disk) surfaces a spurious change on the
   first rebuild. Whether the first export is already a fixpoint is **not a
   function of origin format**: legacy origins get deterministic migrations on
   their first in-place re-import (drops empty/background page `options`,
   sanitizes svgDefs filter ids), AND some v3 origins ship cached frame
   thumbnails + orphaned media that Penpot GCs on the next in-place re-import.
   So we run export → normalize → in-place re-import → re-export cycles until two
   consecutive exports have an equal semantic hash — capped at 3 cycles
   (`MAX_SETTLE_CYCLES`) with a hard error if it never converges. The hash uses
   the exact `sync-core` normalizer the daemon uses, so "two consecutive exports
   equal" proves the DB's current export is a fixpoint. A clean file converges
   after **1** cycle; a migration/GC-heavy file after **2**. The proven-fixpoint
   state is never re-settled (the loop stops the moment it stabilises).
5. Return the new file's exact `/#/workspace?team-id&file-id&page-id` deep link.

**Display-name cap.** A user-supplied `name` is length-capped at 250 BEFORE the
RPC (`MAX_DISPLAY_NAME_CHARS`, `validate_display_name`), mirroring Penpot
2.16.2's import-binfile `name` schema `[:string {:max 250}]` (verified against
the runtime jar). We count UTF-16 code units to match the backend's Java
`String.length()` exactly, so a name is accepted iff the backend would accept it.
The backend otherwise rejects an over-long name with an opaque 5xx (a 300-char
name previously surfaced as HTTP 502); we return a clean `400 {"error":"name too
long…"}` instead. Derived default names (from the template id) are always short,
so this only ever trips an explicit over-long `name`.

The sync daemon then materializes the `.penpot` tree on disk on its own poll
cycle; because the DB is already settled to a fixpoint, the tree it writes is
its own round-trip fixpoint — so the core invariant (delete DB, rebuild from
disk) surfaces no spurious change.

### Why the old format-gated `needs_settle()` was wrong (the fixed bug)

The earlier code settled **only** LegacyV1 origins (`needs_settle()` returned
true for legacy, false for v3). That silently assumed "v3 origins are a fixpoint
on first export". Two v3-zip templates break that assumption — **tokens-starter-
kit** (10 pages / 548 boards) and **penpot-design-system** (41 pages / 11338
boards, ~37 MB) — because they ship cached thumbnails + orphaned media that
Penpot GCs on the next in-place re-import. Their first on-disk export was
therefore NOT a fixpoint, so an invariant rebuild surfaced a spurious change on
the FIRST rebuild. The other 13 of 15 were unaffected (content was always
correct for all 15; this was purely a settle-gating bug). The fix replaces the
format gate with a fixpoint check — a round-trip hash comparison — which is the
correct gate regardless of format.

## Surfaces built

- **Catalog** (`templates.rs`): magic-byte classification (`classify_magic`),
  display-name derivation (`display_name_from_id`), `enumerate_catalog` (offline,
  reads only the first 4 bytes of each file), sorted + drop-with-reason.
- **`GET /__api/templates`** — the shippable catalog (id, name, format, size).
- **`GET /__templates`** — the gallery page (`templates.html`), plain
  HTML/vanilla JS, fully self-contained (no external asset — offline), same
  pattern as `/__home` & `/__palette`. "New file" → POST → deep-link into the
  workspace.
- **N4 palette stub wired for real** (`palette.html`): the "New from template…"
  verb now opens `/__templates` (was a deferring message → N6). Nothing is
  created from the palette itself — the gallery is where the explicit create
  happens.
- Routes merged into the proxy extra router in `apps/desktop/src/lib.rs::boot`
  (rebuilt per boot, so `team_id` stays fresh across a vault switch).

## Reported numbers

- **Templates shipped:** 15 / 15 (4 v3-zip, 11 legacy-v1; all settled to a
  round-trip fixpoint). Zero dropped.
- **Fixpoint safety:** all 15 are now fixpoint-safe (was 13 / 15 before this
  fix — tokens-starter-kit and penpot-design-system were not). Settle cycles to
  converge (observed live 2026-07-15): clean v3 (black-white-mobile-templates)
  **1**; asset-heavy v3 (tokens-starter-kit) **2**; legacy (welcome, ux-notes,
  open-color-scheme) **2**; heavy v3 (penpot-design-system) **2**.
- **New-from-template latency** (POST + settle → materialized + indexed on disk,
  empty vault, renders OFF): black-white-mobile-templates 3p/139b ≈ 14.7 s,
  tokens-starter-kit 10p/548b ≈ 17.4 s, welcome 1p/5b ≈ 8.4 s, ux-notes 3p/98b
  ≈ 7.7 s, open-color-scheme 2p/7b ≈ 6.5 s. The settle-until-fixpoint loop adds
  one export + (n-1) in-place re-imports + n exports on top of the import.
- **Round-trip A=B (materialized-on-disk fixpoint), verified live:**
  - tokens-starter-kit — A == B ==
    `5993f7e4ad3850f70635243102842d899036aa8881dfddee70a6562f600ecc27`
    (settleCycles=2). Previously FAILED A=B; now passes.
  - penpot-design-system — A == B (gate PASS, roundTripAB true, settleCycles=2),
    41p / 11338 boards.
  - Plus the three legacy legs and the clean v3 leg, all A=B.
- **penpot-design-system cost (why it is opt-in):** ~37 MB / 41 pages / 11338
  boards. A single new-from-template with the settle loop took **≈ 112 s** live
  (POST→materialized), dominated by two ~30 s+ export/import round-trips of the
  full 37 MB binfile. It is verified live (N6_INCLUDE_HEAVY=1) but kept OUT of
  the default gate so the fast gate stays ~1 min of template work.

## What is gated where

- **Headlessly gateable (this milestone), default fast run:** catalog listing,
  offline gallery, new-from-template end-to-end (disk materialization + content
  match + A=B round trip) for a **clean v3** (black-white-mobile-templates), the
  **asset-heavy v3** whose first export is not a fixpoint
  (**tokens-starter-kit** — added to close the settle-gating blind spot), and
  **three legacy** origins (migration settle path), plus invalid-id handling.
  The `tokens-starter-kit` leg means the "v3 is always a fixpoint" assumption
  can never silently regress again.
- **Opt-in heavy leg (`N6_INCLUDE_HEAVY=1`):** adds **penpot-design-system**
  (37 MB / 11338 boards) to the round-trip A=B check. Off by default because a
  single settle+round-trip on it takes ~2 min; verified live at least once (see
  Reported numbers).
- **Orchestrator (packaged ladder):** a fresh dmg under `env -i` + poisoned
  proxies boots offline to an empty vault's lighttable and creates a working
  file from a template; `n1..n5` + `just e2e` against the packaged artifact.
