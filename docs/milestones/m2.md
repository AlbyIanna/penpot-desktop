# M2 — One-way sync (DB → FS) + startup reconciliation

## Goal (from PLAN.md)

Direction A: poll `get-project-files`, debounce, `export-binfile` (v3), unzip, normalize,
hash-ledger no-op detection, two-phase dir swap — plus startup reconciliation (on disk but
not in DB → import; in DB but not on disk → export). Ship the manifest
(`.penpot-sync.json`: fileId ↔ path ↔ revn ↔ lastSyncedHash).

## Exit criteria (from PLAN.md)

> The core-invariant test passes: `rm -rf` the Postgres data dir, restart, all files
> restored from disk.

A failure of this test is a **P0** (CLAUDE.md core invariant: the user's folder tree is the
source of truth; the Penpot database is a disposable cache).

## Component checklist

- [x] **`crates/sync-core`**: normalization (sorted keys, 2-space indent, non-ASCII
      preserved, LF, trailing newline), volatile-key stripping (`createdAt`/`modifiedAt`),
      semantic tree hash (sha256 over sorted `(relpath, content-sha256)` pairs — byte-equal
      to `scripts/roundtrip.py`, verified against CPython), versioned manifest with atomic
      save, two-phase dir swap + orphan cleanup. Trees are compared, never zip containers.
- [x] **`crates/sync-daemon`**: poll loop (2 s) on `get-project-files`; change =
      `(revn, modifiedAt)` differs from last synced; per-file 3 s debounce (reset only on
      further change); export pipeline `export-binfile (embed-assets=true,
      include-libraries=false)` → unzip → normalize → semantic hash → no-op discard (never
      touches target mtimes) → manifest hash saved *before* `commit_dir_swap`. Failed polls
      skip the cycle (never treated as deletions); vanished files are left on disk.
- [x] **Startup reconciliation** (`engine.rs::reconcile`, runs before the poll loop,
      retried forever): cleanup orphans → snapshot (with retry) → disk walk → pure decision
      table (`plan.rs`, exhaustive unit tests) → forgets, imports (disk→DB), exports
      (DB→disk), no-op seeding. Per-file failures are logged and skipped so one broken file
      cannot wedge the daemon.
- [x] **FS→DB restore path**: in-place `import-binfile` (kebab-case multipart, `file-id`)
      preserves the file UUID. **2.16.2 gotcha discovered live**: in-place import onto a
      *nonexistent* file-id fails (SSE `error` event, `object-not-found`) — the working
      resurrect recipe is `create-file` with the client-chosen old uuid (`id` is an optional
      schema param) followed by in-place import onto it. `create-file` with a soft-deleted
      id 500s (~7-day retention window) → falls back to import-as-new (manifest re-keyed,
      logged loudly).
- [x] **Retry policy** (`retry.rs`): transient = transport errors + 5xx; 10 attempts,
      0.5 s → 15 s cap (~90 s total) — rides out the 30–60 s backend respawn window.
- [x] **Wiring** (`apps/desktop/src/lib.rs`): daemon spawned after the proxy (export
      downloads go through the proxy's public URI), authed with the `credentials.json`
      token + `profile.defaultTeamId`; sync root `<data_dir>/designs`
      (`PENPOT_LOCAL_DESIGNS_DIR` override); `RunningApp::shutdown` stops the daemon before
      proxy/supervisor.
- [x] **Conflict arm placeholder**: both sides diverged from `lastSyncedHash` → loud
      `tracing::error` + `TODO(M3 conflict rule)`, DB wins for now, disk is never silently
      clobbered on the reconciliation import path. Full conflict copies are M3.
- [x] **Core-invariant test**: `scripts/m2-invariant.sh` (`just invariant`), see below.

## The core-invariant test — `just invariant`

`scripts/m2-invariant.sh` (+ `scripts/m2_invariant_helper.py`, which reuses the
M0-verified hash/normalization functions from `scripts/roundtrip.py`). Dedicated ports
8892/6367/5439/6386, fresh `mktemp` data dir and designs dir; PASS/FAIL per step:

1. **(a)** boot headless; via RPC create 2 projects ("Client A", "Client B") and 3 files
   (alpha, beta in A; gamma in B), each with a rectangle; alpha additionally gets an
   uploaded PNG (`upload-file-media-object`) referenced by an image-filled rect so media
   blobs are covered end-to-end.
2. **(b)** wait until all 3 `.penpot` dirs exist on disk and the manifest is caught up
   (entry `revn` == DB revn *and* `lastSyncedHash` == independently recomputed semantic
   disk hash); assert alpha's tree contains a non-JSON media blob; record per-file hashes.
3. **(c)** SIGTERM → clean exit, no orphan java/valkey/postgres processes.
4. **(d)** `rm -rf <data_dir>/postgres` — surgical: manifest, the 3 `.penpot` dirs,
   `secret.key` and `credentials.json` must survive.
5. **(e)** boot again: provisioning re-registers on the wiped DB; startup reconciliation
   must report `imports=3 exports=0 failed=0`.
6. **(f)** RPC verification: both projects exist, all 3 files exist **with the same
   fileIds** (manifest-driven resurrect recipe), every shape present in `get-file`, media
   blob fetched through `/assets/by-file-media-id/**` byte-identical to the uploaded PNG.
7. **(g)** idempotence: per-file semantic disk hashes identical to (b); no `.conflict-*`
   dirs — reconciliation must not rewrite what it just imported.
8. **(h)** third boot: reconciliation is a pure no-op (`imports=0 exports=0 noops=3
   failed=0`); recursive `stat` fingerprint (name/size/mtime/ctime, manifest included) of
   the designs dir unchanged; DB fingerprint (project names, per-file `revn` +
   `modifiedAt`) unchanged.
9. **(i)** clean shutdown, no orphans, all 4 ports freed.

## Independent verification (2026-07-13)

Re-run from scratch by a separate verification agent (trust-nothing pass), macOS arm64,
Penpot 2.16.2. Grades per exit criterion:

- **Core invariant (`rm -rf <data>/postgres`, restart, everything rebuilt from disk with
  no data loss)** — **MET**: `scripts/m2-invariant.sh` re-run twice back-to-back on fresh
  temp dirs: **17/17 PASS both runs**, `sameIds=True` both times, no flakiness observed.
  Independently reproduced a third time with unicode names (below).
- **`cargo test --workspace` green** — **MET**: 129 passed, 0 failed, 5 ignored
  (live/fuzz), matching the milestone claim.
- **Direction A pipeline (poll → debounce → export → normalize → hash no-op → two-phase
  swap)** — **MET (live)**: `update-file` on a running stack → re-exported tree on disk
  with manifest caught up in **6.8 s** (inside poll 2 s + debounce 3 s + export); an
  UNCHANGED sibling file's recursive stat fingerprint (size/mtime/ctime) was untouched by
  the export.
- **Hash-ledger-before-swap ordering (PLAN.md Direction A step 6, the M3 watcher
  loop-prevention prerequisite)** — **MET (verified in source)**:
  `crates/sync-daemon/src/engine.rs::export_file` inserts the new `lastSyncedHash` and
  calls `manifest.save()` *before* `commit_dir_swap()` (manifest save on the line
  directly above the swap; no-op branch never touches the target).
- **Manifest shipped** — **MET**: `.penpot-sync.json` carries fileId ↔ path ↔ revn ↔
  lastSyncedHash; `lastSyncedHash` verified byte-equal to an independent CPython
  recomputation (`scripts/roundtrip.py` functions) on every synced file in every run.
- **Startup reconciliation, all arms** — **MET (live)**: DB-wipe → `imports=N` under the
  SAME file ids; missed pending change (SIGKILL before debounce fired) → `exports=1` on
  restart with the lost edit re-materialized on disk; empty-designs + empty-DB boot →
  all-zero reconciliation, zero churn (designs dir stat fingerprint stable across ~7 idle
  poll cycles); clean state → pure no-op.
- **M1 not regressed** — **MET**: `bash scripts/m1-smoke.sh` all 9 steps PASS after M2.

Adversarial probes beyond the invariant script (all passed):

- **SIGKILL crash recovery**: SIGKILL right after READY (once mid-export-window, once
  ~1 s after an `update-file` so the pending change died with the process). Restart:
  planted `*.penpot.tmp-*` and `*.penpot.old-*` (target intact) leftovers were both
  swept (`removed_tmp=1 removed_old=1 restored=0`), reconciliation recovered the missed
  edit (`exports=1`), no data loss, no leftovers after.
- **Unicode names**: project `Progetto è🎨`, file `Schizzo è🎨` → sane on-disk path
  `Progetto è🎨/Schizzo è🎨.penpot` (non-ASCII preserved raw in paths *and* inside the
  normalized JSON — zero `\u` escapes, trailing LF verified); full wipe cycle
  resurrected both files under the same ids with disk hashes byte-identical.
- **Known hazard (not an M2 exit criterion, carried to M3/M4)**: SIGKILL of the app
  orphans all three children (postgres/valkey/java keep running and hold their ports) —
  there is no watchdog, so a restart cannot bind until they are killed manually, and an
  orphaned backend keeps accepting writes. Already flagged in m1.md's implications;
  became observable-by-test here.

## Implications for M3

- The resurrect recipe's soft-deleted-id corner (`create-file` with a soft-deleted uuid
  500s) currently burns ~90 s of retries before falling back to import-as-new — probe
  `get-file` first or shrink that retry budget.
- `plan.rs` conflict arm carries the `TODO(M3 conflict rule)` marker — that is where
  `.conflict-<timestamp>.penpot` copies belong.
- Direction B (FS→DB on file change) can reuse the reconciliation import pipeline as-is;
  loop prevention needs the manifest hash checked *before* importing (already how
  reconciliation decides no-op).

### Additions from independent verification

- **Watcher loop prevention is ready**: the hash is in the ledger before the swap lands
  (verified in `engine.rs::export_file`), so the M3 `notify` watcher can compare the
  event-tree's semantic hash against `lastSyncedHash` and skip our own writes. The
  remaining M3 work is only the watcher + debounce + the import call.
- **Conflict-rule TODO locations** (all carry `TODO(M3 conflict rule)`):
  `crates/sync-daemon/src/plan.rs:41` (decision-table arm `ExportDbWinsConflict`),
  `crates/sync-daemon/src/engine.rs:271` (the loud `tracing::error` where the
  `.conflict-<ts>.penpot` copy must be written), `crates/sync-daemon/src/lib.rs:22`
  (module docs).
- **PLAN.md should learn** (risk 1 / Direction B step 4): on 2.16.2, in-place
  `import-binfile` onto a *nonexistent* file-id fails (`object-not-found`) — it only
  replaces existing files. The working restore path is `create-file` with the
  client-chosen old uuid, then in-place import onto it; `create-file` 500s if the id is
  held by a soft-deleted row (~7-day retention) → import-as-new fallback.
- **Hard-crash orphan children**: add a child watchdog (process-group kill / kqueue
  `EVFILT_PROC`) and a boot-time check for a stale postgres/valkey/backend squatting our
  ports+data dir — with two-way sync an orphaned backend accepting writes is a real
  data-consistency hazard, not just a port squat.
