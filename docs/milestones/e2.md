The spine's on-disk half (PLAN3.md chapter 3): the in-vault package home, a
git-diffable per-vault lockfile, and a generalized installer. A package is now
**just a folder / git repo** you clone into your vault; installing it is an
**explicit verb** that lands it as an ordinary vault `.penpot` file — so the
proven M2 core invariant carries it across a database wipe with no new machinery.

## What E2 builds on (reuse, not reinvent)

- **Installer = N6's `templates.rs`.** The import-as-new + settle-until-fixpoint
  mechanism was factored out of `templates.rs::import_new_from_template` into a
  shared `apps/desktop/src/installer.rs` (`default_project_id`,
  `import_binfile_and_settle`, `settle_to_fixpoint`, `first_page_id`). Templates
  (package-type #0) and packages now call the SAME path; the only difference is
  the import source (a template is shipped binfile bytes with a format-dependent
  `version`; a package is a normalized `.penpot` tree we `zip_dir` into a
  version-less v3 binfile). `templates.rs` shrank to a caller of `installer::`.
- **Lockfile = `manifest.rs`.** `crates/sync-core/src/lock.rs` is a near-clone of
  the manifest discipline: versioned schema (`schemaVersion`), hard-fail on an
  unknown version (`Error::LockSchema`), `deny_unknown_fields`, atomic
  tmp+fsync+rename via the shared `util::atomic_write`, byte-diffable
  serialization (sorted keys, 2-space, LF, trailing newline via `normalize::dumps`).
- **Blindness = the existing dot-dir skip.** `.penpot-packages/` is dot-prefixed,
  so the daemon's event watcher (`watcher.rs`) and full reconcile walk
  (`engine.rs::walk_penpot_dirs`) already skipped it. E2 makes that a **named,
  tested invariant**: `sync_core::PACKAGES_DIR_NAME` + explicit defense-in-depth
  guards in all three code paths (`map_event_path`, `is_structural_event`,
  `walk_penpot_dirs`) so the blindness survives any future relaxation of the
  blanket dot rule (e.g. to peek at `.penpot-vault`), plus dedicated unit tests.
- **`contractHash` = E1.** `vault_index::extract_contracts` over the package's
  `.penpot` tree, **file-id excluded**, canonicalized and sha256'd — stable
  across the import id churn because E1 proved the contract body is uuid-invariant
  (`packages::contract_hash_of_tree`; regression test
  `contract_hash_is_stable_across_uuid_churn`).

## The three verbs (headless-triggerable, same-origin through the proxy)

Mirroring N6's `/__templates` precedent (auth cookie + RPC for free):

- `GET  /__api/packages` — every locked package + whether its materialized file
  is currently live in the DB. This is the **DB-wipe re-apply witness**.
- `POST /__api/packages/fetch {url, id?}` — `git clone` a repo into
  `.penpot-packages/<id>` (offline for a `file://` / already-cloned repo;
  idempotent — a present dir is a no-op). `sourceGitUrl` is later read from the
  clone's `git remote get-url origin`.
- `POST /__api/packages/install {id, name?}` — discover the package's single
  `.penpot` source tree, import it as a NEW vault file (shared installer), settle
  to a fixpoint, and write the `lock.json` entry. **Idempotent**: if the lockfile
  already pins `id` at the same `contentHash` AND that `fileId` is still live, the
  install is a no-op (`alreadyInstalled=true`, zero settle cycles).

## The model — why install materializes an ordinary vault file

Installing a design-data package imports its `.penpot` tree **as a new vault
file** (N6 already lands new-from-template in Drafts on disk). It then becomes an
ordinary sync-tracked file, so a DB wipe rebuilds it by the **proven M2
resurrect-by-id reconcile** (in-place import preserves the file-id, invariant 1)
— no bespoke re-import path for the bytes. The lockfile records the pin
(`{version, kind, contentHash, contractHash, sourceGitUrl, fileId}`) so the
re-apply is verifiable and so later types (E3 `libraryShared`, E7 `pluginProps`)
have a stable place to re-derive the DB-only pointers a plain tree cannot carry.
`lock::plan_reapply` is the pure, unit-tested decision (per entry: already
resurrected by id vs. would need re-import from the package source) — for E2
design-data every entry is `AlreadyResurrected`, which is exactly the honest
finding: **E2's re-apply IS M2**, and the lockfile is the ledger that makes it
assertable + carries provenance forward.

## Exit criteria (PLAN3 E2 — verbatim, all met)

- **drops a template and a component-library `.penpot` under `.penpot-packages/`
  and asserts the daemon NEVER enumerates, hashes, or conflict-copies them (edit
  a file inside → no manifest entry, no `.conflict` copy, no import)** — block
  (a): after editing a json inside each package tree and dropping a stray file,
  the sync manifest is byte-unchanged, no manifest path references
  `.penpot-packages/`, the boards count is unchanged (nothing imported/indexed),
  `lock.json` still pins 0 packages (no auto-install), no `.conflict-*` appears. ✓
- **an explicit install imports and settles to a fixpoint (two equal semantic
  hashes — no phantom diff on first rebuild, the N6 P0) and writes a lock entry**
  — block (b): install returns a fresh `fileId`, 64-hex `contentHash` +
  `contractHash`, `settleCycles ≤ 3`; the materialized tree round-trips A=B; the
  `lock.json` entry matches ({version, contentHash, contractHash, sourceGitUrl,
  fileId}). ✓
- **delete-DB + reboot re-applies every locked package deterministically
  (in-place import preserves file-ids, invariant 1) with NO user-disk write
  outside `.penpot-packages/`** — block (d): after `rm -rf <data>/postgres` +
  reboot, every locked `fileId` is live again with the SAME id, `lock.json` is
  byte-identical, the design trees + `.penpot-packages/` are both byte-untouched
  (raw-hash equal), no `.conflict-*` copy. ✓
- **`git clone <url>` lands a package and it installs offline** — block (c): a
  local bare repo (`git clone --bare`) is fetched via a `file://` URL back into
  `.penpot-packages/` and installs with no network; its `sourceGitUrl` is the
  clone origin (a git repo, not a registry). ✓
- **run-twice = no-op** — block (e): a second install of each package is
  `alreadyInstalled=true` with zero settle cycles and leaves `lock.json`
  byte-identical; and the whole gate is run-twice idempotent (fresh dirs). ✓

## Notes / honest scope

- **No DB-only pointers for design-data yet.** `libraryShared` (E3
  `set-file-shared`) and `pluginProps` (E7) are modelled in `LockEntry` but empty
  for an E2 install — the file existence itself is re-derived by M2. The
  `plan_reapply` seam and the boot-time lock reconcile hook exist so E3/E7 fill
  them in without reshaping the lockfile.
- **`fetch` clones once.** A present `.penpot-packages/<id>` is treated as
  already-fetched (idempotent). A real `git fetch`/update-in-place is E3+ scope;
  E2 proves the clone-and-install path.
- **Offline** here means "no registry, no network" — a `file://` clone needs
  neither. The lockfile records the source URL, per the git-repos-not-a-registry
  value.
- The gate authors both packages from the same small clean v3 template
  (`welcome`) for speed; the installer is content-agnostic (the settle is a
  round-trip check, not a format gate), and the asset-heavy/legacy settle paths
  are already covered by the N6 gate the installer was factored from.

## Follow-up fix (cleanup_orphans blindness, post-verification)

Independent verification found the 4th vault-root walker was unguarded: `sync_core::swap.rs`
`cleanup_orphans` (the startup swap-orphan sweep) recurses into dot-dirs (to sweep
`.penpot-sync.json.tmp-*`), so it descended into `.penpot-packages/<id>/` and would
restore/delete any entry matching the swap-orphan grammar `<base>.penpot.{tmp,old}-<12hex>`
inside a package. Impact was confined to `.penpot-packages/` (git-recoverable) and never
crossed into the manifest, design sync, index, lighttable, or a `.conflict` copy — but it
contradicted the daemon-blindness invariant and left 3-of-4 walkers guarded. Fixed: skip
`name == PACKAGES_DIR_NAME` when recursing (mirrors the watcher/walker guards); regression test
`cleanup_never_touches_the_package_home` (a package-internal orphan is left alone while a
genuine root-level orphan is still swept). All four vault-root walkers now guard `.penpot-packages/`.

## Follow-up fix (N5 index-lag flake, surfaced while gating E2 — NOT an E2 regression)

Running the full `just e2e` ladder with E2 present flagged `scripts/n5-vaults.sh`
`(c) SPILL after A→B` (vault B absent from `/boards`, its needles not searchable). It
was **not** cross-vault spill and **not** caused by E2: `dbFileCount:2, sameIds:true`
(the DB held exactly B's files) while only the **index** read empty. Root cause is a
pre-existing test-harness asymmetry. The vault index is a downstream reader of the sync
manifest (a poll-driven diff, `vault_index::Indexer::poll_once`), so it lands **one cycle
after** `wait_synced` proves the disk↔DB manifest contract. The B→A and A→B-again switches
already wait for that eventual-consistency barrier via `wait_present` (which polls until the
needles are searchable); the first A→B switch gated only on `wait_synced`, so `assert_state`
could read `/boards` + `/search` one poll early and flake. Confirmed by a clean-`main`
baseline run (no E2 changes) that N5 is green — and that none of E2's diffs touch the index
path (they gate exclusively on the literal `.penpot-packages` name). Fixed by adding the
same `wait_present B` index barrier on the A→B path in `scripts/n5-vaults.sh`, making all
three post-switch asserts symmetric. The barrier keeps its timeout, so a genuine index hang
would still fail; only the benign one-poll lag is absorbed. The zero-spill invariant itself
was never at risk.
