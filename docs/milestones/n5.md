## What was built

### Vault bookkeeping — `apps/desktop/src/vault.rs`
- **Vault dotfolder** `<vault-root>/.penpot-vault/vault.json` — a stable vault
  id (minted once, reused forever) + a reserved settings map, living INSIDE the
  vault so it travels with a git clone. The disk walker already skips every
  `.`-prefixed dir (`sync-daemon::walk_penpot_dirs`, verified), so the marker is
  never mistaken for a project and never synced. A vault root is *any* directory
  the user points at — empty, an existing tree, or a git clone; `ensure_vault`
  creates it + the marker on first open.
- **Registry** `<data-dir>/vaults.json` — the currently-open vault (`active`) +
  the list of known vaults. Lives in the app data dir, OUTSIDE any vault.
- **Switch marker** `<data-dir>/vault-switch.json` — written (atomically, fsync)
  BEFORE a switch wipes the DB, cleared AFTER the target reconciles. Its
  presence on boot is the sole signal that a switch was interrupted.
- **`reset_disposable_state`** — the M2 "delete the DB, rebuild from disk" reset,
  scoped to the disposable cache: it wipes the Penpot cluster
  (`<data>/postgres/data`) + `.pgpass`, the objects-storage blobs
  (`<data>/assets`) and the vault index (`<data>/vault-index/`), but KEEPS
  `<data>/postgres/install` (the binaries), so a switch is offline and fast (no
  re-download, no re-extract). Wiping `<data>/assets` is what closes the
  media-hygiene gap (see "Media hygiene" below).
- **`decide_startup`** — the pure recovery/selection decision table (unit-tested):
  1. switch marker present → **RecoverForward** to the marker's target (beats
     env + registry: an interrupted switch is always completed, never a hybrid);
  2. `PENPOT_LOCAL_DESIGNS_DIR` explicitly set → that path (this keeps every
     pre-N5 script byte-identical — they always set the env, so they never
     consult the registry);
  3. registry has an active vault → that path (the app remembers the last vault);
  4. otherwise the resolved default (`<data>/designs`).

### The switch mechanism — `apps/desktop/src/control.rs`
`VaultRunner` owns the live `RunningApp` and turns "open a different vault" into
the exact M2 operation:
1. write the switch marker (crash safety) BEFORE anything is wiped;
2. `RunningApp::shutdown()` — sync daemon, index, board-export, proxy and the
   supervised postgres/valkey/JVM all go down cleanly (watchdog `bye`, ports
   freed);
3. `reset_disposable_state` — zero residue of the old vault before the new one
   reconciles;
4. registry: `active = target` (both vaults recorded);
5. `boot(target)` — re-provision, reconcile from the new tree (each file
   re-imported under its ORIGINAL id via the manifest), rebuild the index;
6. clear the marker.

Because the wipe (3) always precedes the reconcile (5), forward-completion of an
interrupted switch is safe from ANY kill point: the previous vault's DB state is
gone first. `boot_active_vault` runs the same resolution + recovery on every
boot (shared by the headless runner and the GUI shell).

### Media hygiene (objects storage) — the `<data>/assets` wipe

**The gap.** Penpot's `fs` objects-storage backend
(`PENPOT_OBJECTS_STORAGE_FS_DIRECTORY`, wired at `apps/desktop/src/lib.rs`
`AppConfig::storage_dir` → `<data>/assets`, and `crates/supervisor/src/lib.rs`)
holds every uploaded media blob on disk. The original `reset_disposable_state`
wiped the DB cluster + index but NOT `<data>/assets`, so a switch left the
previous vault's blobs lingering in the shared cache. Not a strict spill (the DB
rows that reference them are wiped, so nothing surfaces in the DB, boards or
search — `GET /assets/by-file-media-id/<id>` 404s once the `file_media_object`
row is gone), but a real local-first-hygiene gap: unbounded growth and one
vault's raw media bytes physically present on disk while another vault is open.

**Empirical safety check (before touching product code).** Booted a stack on
vault A, created a file with an embedded raster (uploaded an 8×8 PNG via
`upload-file-media-object`, referenced it from an image fill so
`export-binfile` carries the blob), let it sync to disk, confirmed
`GET /assets/by-file-media-id/<id>` served the exact 77 bytes, then MANUALLY
`rm -rf <data>/assets` (0 blobs) and drove a switch away and back (A→B→A). After
the B→A reconcile, `<data>/assets` was repopulated and the same fetch served the
exact 77 bytes again (`bytesMatch=true`). **Verdict: SAFE — reconcile
re-materializes each referenced blob from the `.penpot` on disk.** The mechanism:
the switch re-imports every file in-place under its original id via
`import-binfile`, and because the DB cluster is wiped first there is no
`storage_object` dedup row to suppress the write, so the fs backend re-writes the
blob. The `file_media_object` id is preserved across the round trip (verified),
so the `.penpot` blob path — and thus the raw-byte tree hash — is stable too.

**The fix.** `reset_disposable_state` now also wipes `<data>/assets` (keeping
`<data>/postgres/install`). Safe because of the re-materialization above; it
closes the growth/hygiene gap and makes the previous vault's blobs physically
absent while another vault is open. Covered by the `vault.rs` unit test
(`reset_disposable_state_wipes_cluster_assets_and_index_keeps_install`) and, at
the stack level, by the gate's media legs.

**Gate coverage** (`scripts/n5-vaults.sh` + `n5_vaults_helper.py`). Both vault
fixtures now embed a DISTINCT uploaded PNG (referenced by an image fill). After
EVERY switch (A→B, B→A, the second A→B rebuild, and the crash-recovery reboot)
the gate asserts, and all pass:
- **MEDIA zero-spill (served)** — the other vault's media id is NOT served under
  the active vault (its DB rows are wiped);
- **MEDIA zero-spill (disk)** — the other vault's blob bytes are GONE from
  `<data>/assets` (a `cmp` byte-scan finds no match), proving the wipe;
- **MEDIA survives** — the active vault's image is served again with its EXACT
  uploaded bytes, and its blob is re-materialized on disk from the `.penpot`.

The `(e)` byte-identical `.penpot` tree hashes (now including the media blob)
stay identical across A→B→A, proving the embedded raster round-trips
byte-for-byte through the wipe→reconcile.

### Headless control surface (test/automation) — `control.rs` + `bin/headless.rs`
When `PENPOT_LOCAL_CONTROL_PORT` is set, a localhost-only HTTP server exposes the
switch mechanism WITHOUT the GUI dialog, so `scripts/n5-vaults.sh` can drive it:
- `GET /active` → `{id, path}` of the open vault;
- `GET /list` → `{active, vaults:[…]}`;
- `POST /open {path}` → perform the switch, blocking until the target stack is up
  (200 with the new active, or 5xx with the error — the marker is left in place
  on a boot failure so the next boot recovers).
Unset ⇒ no control server (every pre-N5 flow is byte-identical; nothing new
binds). `PENPOT_LOCAL_SWITCH_TEST_DELAY_MS` (test-only) widens the SIGKILL window
by pausing right after the wipe, so the crash test lands deterministically.

### GUI surfaces (manual-QA — macOS-automation-limited, PLAN2 design item 4)
- **Boot through `boot_active_vault`** (`apps/desktop/src/main.rs`): the GUI
  resolves the active vault (registry + interrupted-switch recovery) before
  booting, so the app remembers your last vault and self-heals a crashed switch.
- **First-run**: with no registry, the resolved default (`PENPOT_LOCAL_DESIGNS_DIR`
  or `<data>/designs`) becomes the initial vault and is recorded; the tray "Open
  Vault…" verb switches from there.
- **`File > Open Vault`** is wired as the tray **"Open Vault…"** entry
  (`tray/model.rs` + `tray/mod.rs`): a native `choose folder` picker
  (`dialog::choose_folder`, osascript) → `VaultRunner::switch_to` → the tray
  rebinds to the new stack and the window reloads onto the new vault's `/__home`.
  As with N4's global shortcut, **tauri-driver does not support macOS**, so this
  GUI path is covered by the manual-QA checklist below; the switch MECHANISM
  itself is gated headlessly.

## Exit criteria — `scripts/n5-vaults.sh` (`just n5`)

Ports: proxy 8948, backend 6409, postgres 5483, valkey 6426, control 8949
(exporter 6489 reserved; renders OFF). Fresh mktemp dirs; ANSI-stripped greps;
pg-install cache seeded.

- **(a)** two vaults A and B with OVERLAPPING project names ("Shared"/"Studio")
  and a file each of DISTINCT content (a searchable needle token per file); one
  file per vault ALSO embeds a DISTINCT uploaded raster (a PNG referenced by an
  image fill) so media hygiene is provable.
- **(b)** switch A→B→A driven headlessly via `POST /open {path}` — no GUI dialog;
  `GET /active` confirms the open vault after each switch.
- **(c)** after EVERY switch, ZERO spill: no file of the other vault appears in
  the DB (`get-projects`/`get-project-files`), the lighttable
  (`/__api/vault/boards`), or the index (`/__api/vault/search` for the other
  vault's needle returns 0 hits; this vault's needle is searchable). ALSO MEDIA
  zero-spill + survival: this vault's embedded image is served with its EXACT
  uploaded bytes (re-materialized from the `.penpot`), the other vault's media
  id is NOT served, and the other vault's blob bytes are absent from
  `<data>/assets` (the objects storage is wiped on switch).
- **(d)** every file keeps its ORIGINAL Penpot id across the round trip
  (`sameIds`, `get-project-files`).
- **(e)** both `.penpot` trees are byte-identical before/after (recursive raw-byte
  hash of every file inside a `*.penpot` dir — user disk untouched), no
  `.conflict-*` copies.
- **(f)** per-vault M2-style DB-wipe→rebuild-same-ids proven on BOTH vaults (the
  switch IS that wipe→rebuild; A proven on B→A, B proven on the second A→B).
- **(g)** a mid-switch SIGKILL (killed during the wipe/reconcile window) + reboot
  recovers to a consistent SINGLE vault (the target), marker cleared, only the
  target's files, original ids, both trees untouched, no orphans, ports free.

### Manual-QA checklist (GUI picker — macOS)
1. Fresh install (no `vaults.json`): launch → app opens on the default vault's
   `/__home`; the tray shows **Open Vault…**.
2. Tray → **Open Vault…** → pick an EMPTY folder → the window reloads on an empty
   lighttable; `vaults.json` now lists both vaults with `active` = the new one.
3. Tray → **Open Vault…** → pick a folder that is an existing exported vault (or a
   git clone) → its boards appear; opening a file deep-links into the canvas.
4. Switch back to the first vault → its boards return; the second vault's boards
   are gone (no spill in the grid/palette/search).
5. Quit while a vault is open, relaunch → the app reopens the SAME vault
   (registry `active`), no env var needed.
6. Force-quit (Activity Monitor → Force Quit) during a switch, relaunch → the app
   completes the switch forward to the target (or a consistent single vault),
   never a half-switched hybrid.

## Backward compatibility

Every pre-N5 gate sets `PENPOT_LOCAL_DESIGNS_DIR` and uses a fresh data dir, so
`decide_startup` always takes arm 2 (env wins) — behavior is byte-identical, and
no control server binds unless `PENPOT_LOCAL_CONTROL_PORT` is set. The registry +
markers are additive files in the data dir; the disposable-state reset is only
ever invoked by a switch or its recovery, never by a normal boot.
