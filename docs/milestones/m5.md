# M5 — Freedom features

## Goal (from PLAN.md)

Per-board SVG/PNG auto-export next to sources (needs the exporter service), `git init`
helper + sensible `.gitignore`, "reveal in file manager", file/project
create-rename-move from the OS side reflected in Penpot. Plus the M4 packaging debts
pulled into scope: `tauri-plugin-single-instance`, non-BMP install/data/designs path
pre-flight (PLAN.md risk 8), and `--remap-path-prefix` for release binaries.

## Exit criteria (self-defined, each one an executable block of `scripts/m5-features.sh` — `just m5`)

1. **(a) Auto-export**: boot with exports enabled → every board of every synced file
   gets `<name>.exports/<board>.svg` + `.png` next to the sources (valid XML / PNG
   magic bytes); an edit re-renders **only** the changed file; idle cycles cause
   **zero churn** (whole-tree stat fingerprint, inodes included).
2. **(b) OS rename** of a `.penpot` dir → the **same file id** is renamed in Penpot
   (`rename-file`), revn continuity, no reimport/duplicate.
3. **(c) OS move across project folders** → DB project membership changes
   (`move-files`), same id, `modifiedAt` byte-identical.
4. **(d) Rename while stopped** → startup reconciliation re-keys the manifest under
   the same id (no import-as-new, no orphan re-export at the old path).
5. **(e) `designs-git-init.sh`** → repo + `.gitignore` + README teaching the
   no-overlay lesson; idempotent second run; no-ops gracefully (no commit) on a
   pre-existing repo.
6. **(f) Non-BMP data dir** → clean refusal: fast non-zero exit, clear message naming
   path + character, no crash-loop, no orphans, nothing written.
7. **(g) Single instance**: a second launch against the same data dir never boots a
   second stack and leaves the first unharmed (GUI: plugin focuses the window;
   headless: loud postgres-lock failure — see "honest scope").

## Honest scope statement

- **The exporter is dev-mode only.** It is NOT packaged: it needs a host `node`
  (upstream image pins v24.16.0; v25.8.1 verified working here — re-probe on any
  version bump), the extracted exporter app (`scripts/fetch-penpot.sh`, docker only
  for the one-time extraction) and a playwright-managed chromium
  (`fetch-penpot.sh --with-browsers`, ~500 MB on disk). `PENPOT_LOCAL_EXPORTS=1` is
  **default OFF**; a missing piece fails boot with an actionable message (pre-flight),
  never a crash-loop. Bundling node+chromium re-opens the dmg-size levers from m4.md
  and is future work.
- **Exporter binds 0.0.0.0** (upstream limitation, verified not configurable in the
  compiled bundle) — accepted for dev mode; the packaging phase must firewall or
  patch it.
- **Render floor ≈ 2.2 s per board/format** (upstream renderer has a hardcoded 1 s
  sleep); end-to-end edit→files-on-disk ≈ 20 s through the two poll/debounce layers.
- **Single-instance guard is GUI-only** (`tauri-plugin-single-instance`): a second
  GUI launch focuses the existing window (verified live). The headless bin's answer
  remains the M4-verified loud postgres-lock failure — asserted in block (g). The
  guard keys on the app identifier, so a second GUI launch with a *different* data
  dir is also refused (documented trade-off).
- **Penpot-side renames stay cosmetic** (asymmetry, documented in
  m5-notes-os-rename-move.md): renaming in the Penpot UI does not move directories
  on disk; only the name inside the exported JSON changes. Full DB→FS rename sync is
  future work, deliberately not half-implemented.
- **A dir edited AND moved within one debounce window** degrades safely to
  import-as-new + loud vanish log (hash pairing requires an exact match).
- **Stale `.exports` dirs** of renamed/moved/deleted files are never auto-deleted
  (the service never removes user-visible outputs); prune by hand or with git.
- **NOT attempted (documented debts)**: Developer ID signing + notarization (needs an
  Apple Developer account; the right-click-Open dance from m4.md stands); the NixOS
  VM validation (nix/README.md recipe still pending); exporter **packaging** (above).

## Post-M5 debts (found by independent adversarial verification)

1. **Stale-exporter adoption after an unclean kill.** SIGKILL (or TERM
   escalated to KILL within seconds) during the boot window can orphan the
   exporter child — the deadman watchdog did not reap it (observed live). A
   subsequent boot on the same exporter port then (a) passes the `/readyz`
   probe against the *stale* exporter, (b) its own exporter child dies with
   `EADDRINUSE` and nothing escalates, and (c) every render fails forever with
   `:unable-to-upload-resource` 403 (secret-key mismatch), retried on the 60 s
   cooldown. Sync is unaffected and the tray shows the error, but boot should
   verify the exporter it probed is its own child (or refuse the port).
2. **Shutdown hang while renders are failing.** SIGTERM during a failing
   render batch delayed exit by ~5–7 minutes (observed): the board-export loop
   checks shutdown only between poll cycles, the per-board retry ladder
   (~60 s) is not cancellation-aware, and the non-`biased` `select!` lets the
   ready interval tick keep winning over the shutdown branch. Healthy-path
   shutdown is ~1 s. Fix: `biased` select + pass the shutdown receiver into
   the retry ladder.
3. **Rename+recreate in one window mis-attributes identity (documented
   degradation, sharper edge found).** `mv A B && cp -R other A`: A's file id
   keeps the *path* (in-place import of the new content), B becomes a new id.
   No data loss and the manifest stays consistent, but the id/history follows
   the path, not the content. Additionally the import-as-new file keeps the
   binfile's *internal* name, so `gamma.penpot` can surface in Penpot as
   "beta" with nothing ever reconciling DB name to dir stem for that path.
4. **Milestone scripts' orphan checks are not concurrency-safe.**
   `m4-artifact-test.sh` scans processes/lsof system-wide; a dev stack running
   concurrently produces false FAILs (observed, then 20/20 on a solo re-run).
   Run suites solo, or scope the checks to the stack's own pid tree.
5. **Windows-reserved board names.** `sanitize_stem` handles reserved
   *characters* but passes reserved *device names* (`CON`, `NUL`, `PRN`, …)
   through unchanged — harmless on macOS/Linux, will break exports dirs on a
   future Windows port.
6. **git helper inside an enclosing repo: document the nesting.** Verified
   safe (nested repo, parent history untouched), but files already tracked by
   the parent stay tracked there too, and the parent sees the designs dir as
   an embedded repo. A note in DESIGNS-README.md would save confusion.

## Implications for M6+

- Tray now has the full status surface (sync + exports rows); a richer per-file
  exports sub-menu could subscribe to the same `ExportStatusSnapshot` channel.
- The re-key planner (`plan_rekeys`) is the building block for DB→FS rename sync
  (moving dirs when the user renames in Penpot) if a later milestone wants it.
- An optional "prune stale .exports" action is future work
  (m5-notes-board-export.md).
- Exporter packaging: pin node v24.16.0, bundle chromium headless-shell (93.5 MB
  floor), firewall the 0.0.0.0 bind.
