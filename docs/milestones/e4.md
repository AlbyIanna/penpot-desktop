## What E4 builds on (reuse, not reinvent)

- **The index is N1.** Package rows are just another `kind` in the same FTS5
  `docs` table N1 already filters by the UNINDEXED `kind` column ‚Äî no schema
  change, no `SCHEMA_VERSION` bump (a Package row is additive, so invariant-1
  "delete the index db, rebuild identical" stays intact). `SearchHandle::search`
  is reused *verbatim* with `kind="package"`.
- **The page + deep link are N3.** `/__packages` mirrors N3's framework-free
  `/__home` (an `include_str!` HTML page, vanilla JS, an `a.package-card
  [data-deeplink]` grid) and each hit deep-links via the same
  `vault_index::workspace_deep_link` string the routes-gate asserts on. The E4
  browser leg is `scripts/routes_gate_nav.cjs`'s twin
  (`scripts/e4_gallery_nav.cjs`).
- **The update channel is the N3 strip's shape.** `/__api/packages/updates` is a
  `watch` channel published by a poller with `send_if_modified` (the debounce) ‚Äî
  the exact pattern of the sync-daemon `StatusHub` behind `/__api/vault/strip`.
  There is **no EventSource**; the page would poll on a `setInterval` like the
  strip does. We matched the *behaviour*, not the transport.
- **The contract diff is E1.** A bump is `vault_index::diff_contracts(before,
  after).overall` ‚Äî the pinned "before" reconstructed from the materialized
  vault file (uuid-invariant, so it equals the source-at-install contract), the
  "after" computed fresh over the current `.penpot-packages/<id>` source.
- **The install/lockfile is E2; the conflict rule is the ledger's.**
  `/__api/packages/preserve-drift` reuses `sync_daemon::paths::conflict_path_for`
  + a uniquify-never-overwrite loop (the engine's `stage_to_conflict_copy`
  analog).

## The two decisions worth recording honestly

### 1. The `drop_all_but` keep-set (the critical caveat)

`Indexer::poll_once` garbage-collects any index owner not in the sync manifest
(`drop_all_but(manifest.file_ids)`) ‚Äî that is how an uninstalled/renamed design
file's stale rows disappear. But **packages live under `.penpot-packages/`,
OUTSIDE the sync manifest** (the E2 blindness invariant): a package is never
enumerated by the daemon, so its `pkg:<id>` rows are *not* in the manifest keep
set and would be GC'd on **every idle poll**, emptying the gallery.

The fix: package rows are keyed `pkg:<id>` (a namespace disjoint from the bare
file-uuid owners), and `poll_packages` runs on *every* poll (before the manifest
branch), driven by **`lock.json` ‚Äî not the sync manifest**. It returns the set
of `pkg:<id>` owner ids, and `poll_once` **unions that into the keep-set** before
`drop_all_but`: `keep = manifest.files ‚à™ package_keep`. So:

- an installed package survives every idle poll (proven by
  `packages_indexed_survive_manifest_gc_and_drop_on_uninstall`);
- removing a package from `lock.json` drops it from the keep-set ‚Üí next
  `drop_all_but` GCs its row (the uninstall path);
- a malformed/forward-schema `lock.json` returns the *already-recorded* `pkg:*`
  owners as the keep-set, so a bad lockfile never empties the gallery.

The live gate proves the disk half of this twice: after a **delete-index-db +
rebuild** the package rows come back byte-identical (leg 5), and 200 synthetic
locked packages index and survive alongside the real one (leg 1).

### 2. No real SSE ‚Äî a watch-channel poll, on purpose

PLAN3 phrases E4b as "surface ‚Ä¶ in the status **SSE**", but the honest surface
in this codebase is **not** an EventSource. N3's activity strip is already a
`watch` channel + a client `setInterval` poll (`/__api/vault/strip`), and E4b
matches that exactly: `spawn_updates_poller` recomputes `compute_updates_model`
every 2 s inside a `spawn_blocking`, publishes to a `tokio::watch` channel with
`send_if_modified` (an unchanged model never republishes ‚Äî the debounce), and
`GET /__api/packages/updates` just borrows the current value (instant). A push
transport would be more machinery for no behavioural gain; the gate asserts the
bump appears within the **poll + debounce window**, which is what a consumer
observes either way.

## Surface, don't apply ‚Äî the core value made real

The whole point of E4b is that a dependency's update is **surfaced, never
applied**: the consumer's materialized `.penpot` on disk must stay
**byte-unchanged** while the channel reports a bump. `compute_updates_model` and
its poller are strictly read-only (`Lockfile::load`, `Manifest::load`,
`read_tree`, `diff_contracts` ‚Äî no write path anywhere). The gate proves it
end-to-end (leg 3): it snapshots the raw byte-hash of the materialized consumer
file, edits the **package source** to add an exported color (a MINOR grow) and
then remove one (a MAJOR removal), watches the poll surface `bump=minor` then
`bump=major` with `pinnedContractHash != liveContractHash`, and asserts the
consumer file's raw hash is **identical** across both ‚Äî the update exists as a
diff, it was never re-imported over the file. Applying an update would need an
explicit re-install verb (out of scope); the channel only tells you a bump
exists.

Drift of a managed package (leg 4) reuses the conflict rule verbatim:
`preserve_package_drift` writes the incoming (drifted) source as a uniquified
`<stem>.conflict-<ts>.penpot` sibling and normalizes it, **overwriting neither**
the installed file nor the source ‚Äî both are byte-unchanged after the copy.

## The gate (`scripts/e4-gallery.sh`, `just e4`)

Dedicated ports proxy 8986 / backend 6447 / postgres 5521 / valkey 6464 /
control 8987 (exporter 6527 reserved; renders OFF). House style: fresh mktemp
dirs, ANSI-stripped log greps, pg-install cache seeded, dirs kept on failure,
run-twice idempotent. It authors a managed component-library package (welcome
template + the E3 `add_library_content` RPC for a real, round-trippable **Brand
Teal** exported color), installs it (E2 import + A=B settle fixpoint), then:

1. **(2) deep link** ‚Äî the bundled offline chromium loads `/__packages`, reads
   the target package's `deepLink` from the `/__api/packages/search` payload,
   asserts the matching card's `data-deeplink` equals it, clicks, and asserts
   the landed URL is exactly `BASE + deepLink` (a `/#/workspace?team-id&file-id`
   string) ‚Äî routes-gate style, on navigation *commit*.
2. **(3) update surface** ‚Äî a source edit surfaces the correct MINOR then MAJOR
   bump within the poll+debounce window while the consumer `.penpot` stays
   raw-hash-identical.
3. **(4) drift** ‚Äî `preserve-drift` writes a `.conflict-<ts>` copy overwriting
   neither side.
4. **(5) rebuild** ‚Äî delete the index sqlite, reboot, and the package rows come
   back byte-identical (invariant 1).
5. **(1) torture scale** ‚Äî 200 synthetic locked packages are seeded into
   `lock.json`; the FTS gallery indexes them and term-search returns the correct
   id for the first / middle / last package in **~3 ms each** (well under the
   100 ms budget). *Note:* `/__api/packages/search` caps its returned listing at
   `MAX_LIMIT=200`, so the empty-query count tops out at the cap even though the
   index holds 201 rows; the three range-spanning probes are the completeness
   witness.

The **`g4`** leg in `scripts/m4-artifact-test.sh` proves the same surface ships
in the dmg: under `env -i` + poisoned proxies it authors a package in the
packaged app's own vault, installs it offline, and asserts `/__packages` serves
and `/__api/packages/search?q=m4pkg` returns the seeded package by id with an
exact `/#/workspace` deep link ‚Äî same harness, no new boot machinery. (Written
here; the dmg suite is run by the orchestrator.)

## Post-review fixes (adversarial review; both verified against the code)

Two findings from the rebuild-idempotency / surface-not-apply lenses; the
index-keepset-invariant-1 lens came back clean.

- **(medium) Bump-severity baseline could skew under a drifted install.** The
  lockfile pins only the contract *hash*, not its *body*, so
  `diff_contracts(pinned, current)` is uncomputable and the "before" is
  reconstructed from the installed vault file. That file is sync-tracked, so a
  local edit to a managed package would drift its contract and skew the surfaced
  bump (e.g. a false MAJOR). Fixed: trust the reconstructed baseline only when it
  still hashes to the pin (`contract_hash_of_lib(materialized) ==
  entry.contract_hash` ‚Äî uuid-invariant, so this detects contract-level drift);
  otherwise surface the update **without** a (potentially wrong) severity rather
  than a misleading one. The availability signal (pin hash vs live source hash)
  was always exact; only the severity label was at risk. Exact severity under a
  drifted install would require pinning the contract *body* in `lock.json` ‚Äî a
  deliberate future refinement.
- **(low) `preserve-drift` wrote a copy unconditionally.** Calling it with
  nothing drifted still materialized a spurious `.conflict-<ts>` copy. Fixed with
  a drift gate: no-op (no copy) when the source contract equals the pin;
  snapshot-per-call behavior is preserved only when there *is* real drift.

## Deliberately out of scope

- **Applying** an update (a re-install verb) ‚Äî E4 only surfaces the bump.
- Token packages (E5) and cross-vault library id-remap (E6) are separate spikes.
- No verified tier / badge / rank / price / promotion anywhere in the gallery ‚Äî
  every card renders identically (the ecosystem value is a uniform flat browse
