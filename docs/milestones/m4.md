# M4 — Packaging

## Goal (from PLAN.md)

jlink-minimized JRE, pinned Penpot release fetch, AppImage + dmg, Nix flake (dev shell +
package). Orchestration addition: the packaged app's FIRST boot must be fully offline —
in dev, `postgresql_embedded` downloads postgres binaries from GitHub on first run; the
bundle must pre-seed them.

## Exit criteria (from PLAN.md)

A fresh machine (or clean NixOS VM) runs the app from a single artifact.

## Dev-mode protection (why the resources overlay exists)

`tauri-build` copies `bundle.resources` next to the compiled binary so `tauri dev` can
resolve them. If the resources lived in the base `tauri.conf.json`, every **debug**
build would drop a `penpot-runtime/` into `target/debug/`, and the executable-adjacent
bundle discovery in `layout.rs` would silently flip every dev run (`cargo run`,
m1/m2/m3 scripts) into packaged mode. The overlay config is applied only by
`scripts/build-dmg.sh`; verified after a debug build that `target/debug/penpot-runtime`
does not exist. (The release build does get `target/release/penpot-runtime` — that copy
is cleared and recreated by the build script; see fix #4 below.)

## Failures found and fixed on the way

1. **`build-runtime-bundle.sh` exited 1 on its skip path**: the EXIT trap's last
   command (`[[ -n "$STAGING" ]] && rm`) fails under `set -e` when staging is empty.
   Fixed with an explicit `return 0` in `cleanup()`.
2. **dmg named `x64` on an arm64 machine**: `cargo-tauri` was installed as an x86_64
   (Rosetta) binary and names the dmg after its own arch. Reinstalled tauri-cli 2.11.4
   native arm64 → `Penpot Local_0.1.0_aarch64.dmg`.
3. **`bundle_dmg.sh` failed headless**: the dmg step drives Finder via AppleScript for
   window layout. `CI=true` (in `build-dmg.sh`) passes `--skip-jenkins`, skipping the
   AppleScript.
4. **Second build: "Permission denied (os error 13)"**: tauri-build re-copies resources
   over `target/release/penpot-runtime`, and `std::fs::copy` cannot overwrite the
   bundle's 34 read-only files (jre legal docs, license texts). `build-dmg.sh` clears
   the stale copy first.
5. **Real product bug — env proxies hijacked loopback RPC** (found by the
   poisoned-proxy run): `reqwest::Client::new()` honors `http_proxy` & co., so on any
   machine with a proxy configured the app's own `login`/`register-profile` calls to
   `127.0.0.1` would go through the proxy and first boot would fail. Fixed:
   `PenpotClient` now builds its client with `.no_proxy()` (it only ever talks to the
   local backend). The Java backend, valkey, postgres and the supervisor's raw-TCP
   probes were already immune.
6. **App was only linker-signed**: Tauri does not sign without an identity; set
   `signingIdentity: "-"` → proper ad-hoc bundle signature
   (`codesign --verify --deep --strict` passes, identifier
   `dev.albyianna.penpot-local`).
7. **Test-harness lessons**: `/__bootstrap` is single-use per boot and the GUI webview
   wins the race — readiness polling must use `GET /`; the watchdog's spawned path is
   non-canonicalized (`Contents/MacOS/../Resources/...`), assert on suffix.

## Gatekeeper / notarization (documented consequence)

M4 ships ad-hoc signed, NOT notarized. On any other Mac, a downloaded dmg carries the
quarantine attribute and double-click-open is blocked ("cannot verify the developer").
First launch requires **right-click → Open** (twice on newer macOS: System Settings →
Privacy & Security → "Open Anyway"), or `xattr -d com.apple.quarantine`. A Developer ID
+ notarization is the M5+ answer. Local builds (this machine) have no quarantine bit,
so the artifact test cannot and does not exercise Gatekeeper itself.

## Honest platform coverage statement

**Verified here: macOS arm64 (Apple Silicon) only.** The dmg is a thin arm64 build made
and tested on the dev machine (macOS 15/Darwin 25). The "fresh machine" is an
*approximation* — sanitized env, fresh HOME, poisoned proxies, provenance lsof audit —
not a literal separate machine; SIP-protected system frameworks, the WindowServer
session, and the logged-in GUI user are shared with the dev box, and Gatekeeper's
quarantine flow was reasoned about, not executed. Intel macOS (x86_64) remains
**not** built or tested. The Linux AppImage (x86_64) and the Nix flake ARE now
built and tested — in CI on GitHub runners (see "CI evidence" below): the
AppImage's headless smoke runs against the artifact's own extracted resources
on a runner that never had the dev toolchain (a genuinely fresh-machine-shaped
environment), and a full-GUI boot under xvfb served the SPA as a best-effort
extra. The exit criterion "a fresh machine runs the app from a single artifact"
is met for Apple Silicon Macs (macOS ≥ 15.4 — see CI notes — plus the
right-click-Open Gatekeeper step) and for x86_64 desktop Linux
(headless-proven; FUSE + host excludelist libs assumed, as for any AppImage).
A clean NixOS VM run of the nix package remains future work (needs the Penpot
runtime fetched separately + nix-ld for embedded postgres — nix/README.md).

## Independent verification (2026-07-13, this machine)

Everything below was re-established from scratch, trusting no prior artifact:
`dist/` deleted entirely, `build-runtime-bundle.sh` rebuilt the bundle fresh
(P1–P7 all green, 562.7 MB, fingerprint `penpot=2.16.2 jdk=26.0.1 valkey=9.1.0
imagemagick=7.1.2-27 postgres=15.18.0 node=absent`), old dmg deleted,
`build-dmg.sh` rebuilt it (386 MB), `m4-artifact-test.sh` on the fresh dmg:
**19/19 PASS** — offline first boot READY in 14 s under poisoned proxies,
6/6 `source=bundle`, 0 lsof opens under `/opt/homebrew`, 0 under the repo,
SIGTERM clean, SIGKILL → watchdog reap. Dev regression after all of it:
`cargo test --workspace` 205/0/8, `m1-smoke` ALL PASS, `m2-invariant`
ALL PASS (sameIds=True), `m3-sync` ALL PASS.

Adversarial findings (the honest part):

1. **NOT MET (edge): install paths containing non-BMP characters break the
   backend.** From `/tmp/Applicazioni è🎨/Penpot Local.app` the backend
   crash-loops with `Could not find or load main class clojure.main` and boot
   fails (loudly — "boot failed" window; watchdog still cleans up; nothing
   orphaned). Bisected with a minimal jar: **JDK 26 cannot load classes from a
   jar whose path contains a non-BMP character** (emoji, surrogate-pair) —
   reproduced with the homebrew JDK too, from any cwd, with and without
   `LANG`, with `-jar` and `-cp`; `sun.jnu.encoding` is already UTF-8. Spaces
   + accented BMP unicode (`/tmp/Applicazioni è/`) work fully: READY 13 s,
   6/6 `source=bundle`, auth RPC, SIGKILL→reap all verified from that path.
   Upstream JDK limitation inherited, not a relocation bug; real-world macOS
   install paths (`/Applications`) are unaffected. M5 candidate: pre-flight
   the resolved backend path for non-BMP chars and surface a clear error.
2. **Double launch: no single-instance guard; failure is sane.** A second
   identical instance boots its own supervisor, postgres refuses the shared
   data dir within ~0.3 s (`postmaster.pid` lock), boot aborts loudly, the
   second watchdog exits (no children), and the second window sits on "boot
   failed (see logs)". The first instance stayed fully healthy throughout
   (GET / 200 + authenticated get-profile verified while both ran); the
   second instance exits cleanly on SIGTERM. UX gap, not a data hazard —
   tauri-plugin-single-instance (focus the existing window) is the M5 fix.
3. **Case-sensitive APFS volume: PASS.** dmg residing on the volume mounts;
   the app installed on it boots (READY 17 s), 6/6 `source=bundle`, SPA +
   authenticated RPC, clean SIGTERM — no case-sensitivity assumptions.
4. **Packaged-mode SIGKILL: PASS** (repeated outside the artifact test, from
   the unicode path): watchdog reaped java/valkey/postgres in ~1 s, ports
   freed, zero survivors.
5. **Secrets/dev-droppings sweep: no secrets; one cosmetic leak.** No
   credentials, tokens, `.env`, or keys anywhere in the .app or bundle; the
   penpot-runtime tree has zero build-machine paths; the bundled
   `penpot-watchdog` is clean. But the `penpot-desktop`/`headless` release
   binaries embed ~1087 build-machine path strings
   (`/Users/<user>/.cargo/registry/...` panic/tracing metadata + one
   compiled-in `CARGO_MANIFEST_DIR` dev-fallback path). Username disclosure
   only — standard rustc behavior; fix with `--remap-path-prefix` (or cargo
   `trim-paths` when stable) in M5.
6. `codesign --verify --deep --strict` passes on the installs at both the
   unicode path and the case-sensitive volume.

## Implications for M5

- **Non-BMP (emoji) install-path edge** (finding 1): decide between a boot
  pre-flight with a friendly error vs. copying the runtime to an ASCII path
  under the data dir. Also untested: emoji in the *data dir* / Designs root
  (the sync daemon + java storage paths) — probe it before promising anything.
- **Single-instance UX** (finding 2): add tauri-plugin-single-instance so a
  second launch focuses the window instead of opening a "boot failed" shell.
  The data-dir postgres lock already prevents corruption; this is polish.
- **Strip build-machine paths from release binaries** (finding 5):
  `RUSTFLAGS=--remap-path-prefix` (or `trim-paths`) in `build-dmg.sh` + CI.
- **Notarization**: Developer ID + notarization replaces the right-click-Open
  dance; the ad-hoc signature survives relocation (verified) so the pipeline
  only needs identity + `notarytool` steps.
- **NixOS**: the honest exit-criterion gap. Teach the supervisor to consume a
  nixpkgs postgres (the bundle resolution layer already models pre-seeded
  installs), pull the Penpot images via `dockerTools.pullImage`, then run the
  nix/README.md VM recipe on a clean NixOS VM.
- **Exporter service** (already flagged in the bundle script): bundling it
  brings node (pin v24.16.0 per the upstream image) + puppeteer/chromium —
  expect the dmg/AppImage size conversation (source maps 55.9 MB,
  images/features 90.2 MB, builtin-templates 93.7 MB are the existing levers).
- **Re-run the no-node probe on any PENPOT_VERSION bump** (bundle script
  header documents this) — the node-free bundle is only proven for 2.16.2.
