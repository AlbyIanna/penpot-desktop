# M3 — Two-way sync + conflicts

## Goal (from PLAN.md)

Direction B (filesystem → DB), loop prevention, conflict copies, and error surfacing in a
small status UI (tray/menubar: last sync, per-file state, pause button).

## Exit criteria (from PLAN.md)

> `git checkout` an older version of a file directory → it appears in Penpot within
> seconds; simultaneous-edit test produces a conflict copy, never data loss.

The conflict rule (CLAUDE.md, non-negotiable): never silently overwrite either side — if
both the DB and the filesystem changed since `lastSyncedHash`, export the DB version as a
`<name>.conflict-<timestamp>.penpot/` copy next to the file and surface it.

## Component checklist

- [x] **Direction B, watcher loop** (`crates/sync-daemon/src/watcher.rs` + reworked
      `engine.rs`): recursive `notify` (v8) watcher on the sync root; events mapped to the
      owning `<project>/<name>.penpot` dir; quiescence-based per-dir debounce
      (`SyncConfig::fs_debounce`, default 2 s — editor/`git checkout` event storms fire
      once). On fire: semantic tree hash vs `lastSyncedHash` → own-write/no-op skip
      (**loop prevention**; Direction A saves the hash *before* its dir swap lands, an M2
      guarantee) → validate (`validate.rs`: every `.json` parses, binfile manifest sane;
      failure = per-file error, nothing touched) → deterministic zip → **in-place import**
      (`file-id`, kebab-case multipart) → read back `(revn, modifiedAt)` into manifest +
      poll tracker so our own import never bounces. Unknown dirs → import-as-new. Deleted
      dirs → loud error, DB side never deleted (re-exported at next startup).
- [x] **Conflict rule** (all `TODO(M3 conflict rule)` markers replaced in `plan.rs`,
      `engine.rs`, `lib.rs`): both sides diverged from `lastSyncedHash` → DB version
      exported as `<name>.conflict-<ts>.penpot/` next to the file, disk version imported
      in place under the same file id, manifest fast-forwarded, status = `Conflict`.
      Timestamps are RFC 3339 with `:` → `-` (Finder/Windows-safe), e.g.
      `homepage.conflict-2026-07-13T13-15-26Z.penpot`. Copies are excluded from the
      watcher, the disk walker and reconciliation — never synced, never auto-deleted.
      Applies on the watcher path, the poll path (an export never swaps over a diverged
      disk tree) and the startup-reconciliation arm.
- [x] **Status + control API** (`crates/sync-daemon/src/status.rs`):
      `SyncDaemonHandle::status()` → `watch::Receiver<SyncStatusSnapshot>` (last sync
      time, per-file `FileState` incl. `Conflict{copy_path}`/`Error`, `paused`, sticky
      `last_error`; published only on actual change) and `SyncDaemonHandle::control()` →
      `SyncControl` (`pause`/`resume`/`is_paused`). Pause is coarse and safe: in-flight
      op finishes, then watcher events are dropped and poll cycles skipped; resume
      triggers a full rescan (hash ledger makes clean dirs no-ops) so edits accumulated
      while paused are applied.
- [x] **Tray/menubar UI** (`apps/desktop/src/tray/{mod,model}.rs`,
      `apps/desktop/src/status.rs`): Tauri v2 tray icon with template bitmaps per
      aggregate state (Idle disc / Syncing ring / Paused bars / Attention exclamation);
      menu = relative "Last sync", truncated daemon error row, per-file rows with glyphs
      (conflicts→errors→in-flight→pending→synced, 10-row cap + overflow), pause/resume
      toggle, Quit → `app.exit(0)` → the normal `RunEvent::Exit` clean shutdown. All menu
      logic is a pure unit-tested builder (`model.rs`); the menu is rebuilt on every
      snapshot change + a 30 s tick so the relative time never staleness-drifts.
- [x] **Tray ↔ daemon binding** (`apps/desktop/src/status.rs::DaemonStatusBridge`,
      wired in `apps/desktop/src/main.rs`): the tray is created in Tauri `setup` (main
      thread, before boot) while the daemon exists only after `boot()` — the bridge
      late-binds them: the tray subscribes immediately, the boot task attaches the real
      daemon's status stream + control, and a pause pressed during boot is replayed onto
      the daemon at attach time. Status/control come from the real daemon via
      `RunningApp::sync_status()` / `RunningApp::sync_control()`.
      `PENPOT_LOCAL_TRAY_DEMO=1` keeps the scripted mock (menu QA without a stack).
- [x] **SIGKILL orphan watchdog** (`crates/supervisor/src/{watchdog.rs,bin/penpot-watchdog.rs}`,
      hazard carried from m2.md): the supervisor spawns one `penpot-watchdog` child (own
      process group, signals ignored) fed over a stdin pipe (`grace`/`pgdata`/`pids`/`bye`
      lines; pid set re-fed on respawn). Pipe EOF without `bye` = parent died (incl.
      SIGKILL) → SIGTERM last-known pids, grace (5 s), SIGKILL survivors, then SIGKILL
      anything whose cmdline contains the pgdata path (postmaster re-parents). Clean
      shutdown sends `bye` — the watchdog exits killing nothing. Binary resolution:
      `PENPOT_WATCHDOG_BIN` → config → sibling of the app executable; missing binary =
      loud warning, boot proceeds. Packaging (M4): bundle `penpot-watchdog` next to the
      app executable.
- [x] **Manifest** stays schema v1: `dbModifiedAt` added additively (serde-defaulted, M2
      manifests load unchanged).
- [x] **M1/M2 not regressed**: `scripts/m1-smoke.sh` and `scripts/m2-invariant.sh` fully
      green with the watcher + watchdog active (both scripts now also build the
      `penpot-watchdog` bin — `cargo build -p penpot-desktop` alone does not build
      dependency-crate bins; same note applies to `just dev`).
- [x] **Exit-criteria script** (`scripts/m3-sync.sh` + `scripts/m3_sync_helper.py`,
      `just m3`): both PLAN.md exit criteria as an executable PASS/FAIL test — see
      "The exit-criteria test" below. Ships a **test-only SIGUSR1 hook in the headless
      bin** (toggles `SyncControl` pause/resume, logs `SIGUSR1: sync paused/resumed`;
      the GUI app does not install it — the tray is its control surface).

## The exit-criteria test — `just m3`

`scripts/m3-sync.sh` (+ `scripts/m3_sync_helper.py`, which reuses the M0-verified
hash/normalization functions from `scripts/roundtrip.py`). Dedicated ports
8900/6375/5451/6394, fresh `mktemp` data/designs/work dirs; PASS/FAIL per step
(33 checks total), m2-invariant style:

1. **(a)** boot headless; seed project "M3 Sync" + file "homepage" with a shape via
   RPC; wait until the daemon mirrors it to disk with a consistent manifest (live DB
   revn == manifest revn, `lastSyncedHash` == independently recomputed CPython semantic
   disk hash); record hash v1.
2. **(b)** `git init` the designs root, commit everything (v1).
3. **(c)** RPC edit (second shape) → daemon exports v2 → assert v1 ≠ v2 → commit v2.
4. **(d) EXIT CRITERION 1**: `git checkout --no-overlay <v1> -- <file dir>`
   (`--no-overlay` because pathspec checkout defaults to overlay mode, which does NOT
   delete files added since v1 — the v2 shape's JSON would survive and the "revert"
   would be a merge) → the watcher imports it; poll `get-file` until the v2-only shape
   is gone; **latency measured and reported**; then a fresh `export-binfile` semantic
   hash must equal v1 (full content revert, ids preserved).
5. **(e)** loop prevention around (d): 10 s of idle polling → the ANSI-stripped log's
   sync-op counter (imports + exports, both directions) frozen, designs-dir recursive
   stat fingerprint byte-stable, DB `(revn, modifiedAt)` frozen.
6. **(f) EXIT CRITERION 2** (simultaneous edit): pause via SIGUSR1 → shape renamed
   ON DISK (renormalized to spec) AND a different shape added via RPC → 6 s paused
   soak proves the daemon touches neither side → record the expected DB hash (fresh
   export) → resume → **exactly one** `homepage.conflict-<ts>.penpot` appears
   (latency reported); its semantic hash == the pre-resume DB export (DB version
   preserved); the live dir still hashes to the disk edit and the manifest is
   consistent; `get-file` shows the disk version (renamed shape present, RPC-only
   shape gone); the RPC-only shape's name greps inside the conflict copy —
   **neither version lost**.
7. **(g)** conflict copy is inert: 8 s more idle → op counter frozen, conflict dir
   stat-untouched, still exactly one conflict dir.
8. **(h)** clean shutdown, no orphans, all 4 ports freed; then a SIGKILL-variant
   boot (two-way daemon active, watchdog armed) → SIGKILL → within 30 s no process
   references the data dir and all 4 ports are free.

The script keeps a persistent copy of the embedded-postgres binaries
(`~/.cache/penpot-local/pg-install`, override `M3_SYNC_PG_CACHE`) and seeds each
fresh data dir from it — without this, every run re-downloads via the GitHub API,
which 403s after ~60 unauthenticated calls/hour (observed live: a second
back-to-back run failed to boot until the limit reset).

Adversarial probes beyond the scripts (all passed):

- **Rapid-fire external edits** (10 edits 0.5 s apart to the same file dir): coalesced

  (semantic hash equality via fresh export); zero conflict copies.
- **Invalid JSON dropped into a synced dir**: surfaced as a loud per-file error

- **Editor-style atomic writes** (write tmp sibling + fsync + `rename(2)` over the
  target): exactly one import, DB == disk, no tmp entries leaked into the DB export,
  no leftovers on disk. Bonus finding: an atomic rewrite with byte-different but
  *semantically identical* content produced **zero** imports (the semantic hash ledger
  also suppresses formatting-only churn).
- **SIGKILL during an active import**, both windows: (a) killed mid-flight before the
  import RPC landed → watchdog reaped all children ≤30 s, restart reconciliation
  re-imported the disk edit in place (`imports=1`), DB == disk, no data loss; (b) killed
  in the ~10 ms window *after* the import RPC but *before* the manifest finalize (stale
  `lastSyncedHash` proven on disk) → restart saw "both moved", detected the staged DB
  tree == disk tree and **fast-forwarded the ledger without a spurious conflict copy**
  (`exports=1 conflicts=0`), DB == disk. Nothing vanished in either variant.
- **Pause is airtight under a poll+debounce storm**: see exit criterion 2 above.
- **Unicode dir names through the full git cycle**: see exit criterion 1 above; names
  raw in the normalized JSON (no `\u` escapes).

Minor observation (not a violation, worth a user-docs note in M5): a **foreign file
placed inside a `.penpot` dir** (e.g. a crashed editor's `stray.txt~`) changes the
semantic tree hash, so it triggers an in-place import (Penpot 2.16.2 tolerates the
unknown zip entry) and is then **silently deleted by the next DB→FS export** — the
two-phase swap replaces the whole directory. The design content is never lost, but a
user stashing notes inside a `.penpot` dir will lose them on the next Penpot-side edit.
`.penpot` dirs should be documented as generated, tool-owned trees.

## Implications for M4 / notes for the ExitCriteria phase

- **Timings to expect**: external disk edit → in Penpot ≈ 2.3 s (fs debounce 2 s
  dominates); DB edit → on disk ≈ 4.4 s (poll 2 s + DB debounce 3 s window); conflict
  copy after resume ≈ 0.3 s; startup reconciliation adds seconds on a warm boot.
- **Cosmetic**: after each Direction A export the watcher briefly flips the file
  Pending → Synced (~2 s) while it hash-verifies our own write; the tray shows a short
  "Syncing" blip. Harmless.
- **Soft-deleted-id corner** (from m2.md, still true): resurrecting a file whose old id
  is held by a soft-deleted row burns ~90 s of retries before the import-as-new
  fallback.
- **Deleted dirs**: loud error + `Error` status, DB never deleted; the file is
  re-exported at the next startup reconciliation. Code-pathed but not separately
  live-asserted.
- **Watchdog packaging** (M4): bundle `penpot-watchdog` next to the app executable or
  set `PENPOT_WATCHDOG_BIN`. Dev flows must build it (`just dev` does; the smoke/invariant
  scripts do).
- **Tray in dev**: `cargo run -p penpot-desktop` shows the real tray bound to the real
  daemon; `PENPOT_LOCAL_TRAY_DEMO=1` switches it to the scripted mock frames.
- **Test ports taken so far**: 9001/6060 (m0), 8788/6263/5435/6382 (m1-smoke),
  8892/6367/5439/6386 (m2-invariant), 8894/6369/5443/6388 (M3 direction-B component),
  8990/6470/5440/6390 (tray component), 8899/6377/5449/6399 (watchdog component),
  8898/6373/5447/6392 (`live_app_m3` — baked into the test),
  8900/6375/5451/6394 (m3-sync, `just m3`).
- **Fresh-data-dir boots hit the GitHub API** (embedded-postgres release listing) and
  403 after ~60 unauthenticated calls/hour. `scripts/m3-sync.sh` seeds a persistent
  cache (`~/.cache/penpot-local/pg-install`); the m1/m2 scripts and any new
  fresh-dir test have the same exposure — reuse the cache pattern (or a
  `GITHUB_TOKEN`) if they start flaking. Real user installs are unaffected (one
  download into the persistent data dir, offline forever after).

## Implications for M4 / M5 (from independent verification)

- **Crash-window recovery is solid — keep it that way.** Both SIGKILL windows around an
  import (before the RPC lands; between the RPC and the manifest finalize) converge on
  restart with no data loss and no spurious conflict copy (the staged-tree==disk-tree
  fast-forward arm in `engine.rs::export_file` is what absorbs the second window).
  Any M4 packaging change that alters shutdown ordering or the watchdog's sibling
  lookup (`PENPOT_WATCHDOG_BIN` / bundled next to the executable) must re-run the
  SIGKILL probes, not just the clean-shutdown scripts.
- **Foreign files inside `.penpot` dirs are swept by the next DB→FS export** (two-phase
  swap replaces the whole dir). For M5's `git init` helper + user docs: state plainly
  that `.penpot` dirs are tool-owned generated trees; anything a user adds by hand
  will trigger one import and then vanish on the next Penpot-side edit. If that ever
  needs to be friendlier, options are: warn on non-binfile entries at validate time,
  or preserve unknown files across swaps — decide in M5, don't silently change swap
  semantics.
- **Formatting-only rewrites are free.** A byte-different but semantically identical
  rewrite of a `.json` causes zero syncs. M5's git helper can therefore recommend
  formatters/hooks without fear of sync churn — but only if they preserve the
  normalization spec (a *non*-spec rewrite is a semantic no-op yet still hashes
  equal ONLY after volatile-key stripping and content hashing; key order/indent do
  affect raw bytes on disk and will show up as git diffs).
- **Overlay-mode git checkout is a user foot-gun** (also noted by the ExitCriteria
  phase): plain `git checkout <commit> -- <dir>` does NOT delete files added since
  that commit, so a "revert" of a file that gained shapes imports a merged tree.
  M5's git helper / docs should teach `git checkout --no-overlay` or `git restore
  --source=<commit> --worktree --staged -- <dir>` (verified: `restore` defaults to
  no-overlay and deletes files absent from the source) for reverting `.penpot` dirs.
- **Error surfacing is log+tray only.** The invalid-JSON probe proved errors are loud
  and non-destructive, and healing works — but a headless/scripted user only sees
  them in stderr logs. M5 (or M4's packaging of logs) should decide where persistent
  per-file error state lives (e.g. a `doctor` subcommand or a status file) so a
  broken tree left overnight is not only visible in a tray menu.
- **Test-port ledger addition**: 8902/6377/5453/6396 were used by this verification's
  adversarial probes (one-shot, freed).
