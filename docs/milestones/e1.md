`docs/ecosystem-spikes/contract-extractability.md`) into a real Rust component.
A designer now gets a **"did this edit break my library's contract?"** lint over

## Where it lives ‚Äî extended `vault-index`, no new crate

`crates/vault-index/src/contract.rs`, a sibling of `extract.rs`, exactly as the
PLAN3 Architecture table prescribes. Rationale:

- It reads the **same normalized JSON the ledger hashes** ‚Äî `sync_core::read_tree`
  + `semantic_view`, identical to `extract_docs`. Extending the crate that
  already owns that plumbing (and the reindex loop it will slot into for E4's
  `DocKind::Package`) reuses it verbatim; a new crate would re-import sync-core
  and duplicate the tree-reading seam for no gain.
- Contract extraction is a pure sibling function to search-doc extraction; they
  share the totality discipline (`extract.rs`: unknown paths skipped, missing
  fields default, a malformed tree yields a thinner result, never an error).

Public surface (re-exported from the crate root):

```
extract_contracts(&BTreeMap<relpath, bytes>) -> LibraryContract
diff_contracts(before, after)                -> Classification  // Bump: patch|minor|major|migration
```

plus a thin lint CLI, `src/bin/contract.rs`:

```
contract extract <tree-dir>               # the contract as JSON
contract diff    <before-dir> <after-dir> # OVERALL BUMP: ‚Ä¶ + JSON
contract churn   <src-dir> <dst-dir>      # remap every uuid (import-as-new simulation)
```

## What a contract is

Per **variant set** (the unit a consumer depends on):

| field | source | notes |
|---|---|---|
| `set` (identity) | the shared **`path`** | uuid-free ‚Äî NEVER the `variantId` (caveat 2) |
| `setKind` | `first-class-variant` \| `path-convention` | which of the two models it came from |
| `variantNames` | component `name`s | sorted, deduped; both models |
| `exposedProperties` | distinct `variantProperties[].name` | first-class only; empty for legacy |
| `tokensUsed` | set-union of `appliedTokens` values over the set's main-instance subtrees | sorted, deduped |

plus a **library-level** surface: `exportedColors[]`, `exportedTypographies[]`
(the asset names), and `exportedTokens[]` = the `(path, $type)` leaves of
`tokens.json` (the same dotted vocabulary `appliedTokens` references, set name
excluded).

## The classifier

`patch` = implementation only (`$value` / inline fills / positions changed, the
contract byte-identical). `minor` = the contract grew (a variant / property /
token / exported name **added**, none lost). `major` = an element was
**removed, renamed, or `$type`-changed**. Whole-file bump = the max over every
set plus the library surface.

`$value` vs `$type`: a token's `$value` is implementation (it isn't in the
contract at all, so a `$value`-only edit is `patch`); its `$type` is contract ‚Äî
exported tokens are compared as `(path, $type)` pairs, so a `$type` flip reads
as a rename ‚Üí `major`. A color's hex is likewise implementation; its name is
the contract.

## The three caveats, executed

1. **Two variant models coexist forever.** Components are **grouped** by
   `variantId` when first-class, else by shared `path`; the fixture exercises
   both a first-class set (`Controls / Button`, `variantProperties` = Size/State)
   and a legacy set (`Legacy / Combobox`, names Active/Default/Disabled, empty
   exposedProperties) in one file.

2. **Identity is name/path, never `variantId`.** That uuid is remapped by
   import-as-new. The spike oracle keyed first-class sets on `variantId`; E1 keys
   every set on its uuid-free `path`, so `extract(A) == extract(A')` across a
   settle / uuid churn. The gate proves this by **churning every uuid** in the
   tree (the cheap, stack-free simulation of import-as-new) and asserting the
   contract body is byte-identical and `diff` reads `PATCH` ‚Äî **while the spike
   oracle mis-reads the very same churn as `MAJOR` noise.** That divergence is
   asserted, not hand-waved: it is the concrete hazard E1 fixes.

3. **Legacy‚Üífirst-class migration is not growth.** A legacy set has empty
   `exposedProperties`; migrating it to `variantProperties` (same path, same
   variant names) makes the contract *look* like it grew. `diff_contracts`
   special-cases it: `before.setKind == path-convention && after.setKind ==
   first-class && before.exposedProperties.is_empty()` neutralises the
   exposedProperties delta and labels it **`migration`** (ranked with `patch` ‚Äî
   no consumer break), never a spurious `minor`. Any *genuine* change during the
   migration (a dropped/renamed variant, a dropped token) still bumps normally
   (test `diff_migration_with_dropped_variant_is_major`).

## Oracle parity

The Rust classifier reproduces the python spike oracle
(`scripts/ecosystem-spike/{extract_contract,diff_contracts}.py`) on the spike's
delta matrix EXACTLY. Verified two ways, both executed:

- **`tests/oracle_parity.rs`** (in `cargo test --workspace`) drives the *real*
  python oracle over the authored fixture and asserts rust == oracle == expected
  on `{patch, added‚Üíminor, removed‚Üímajor, renamed‚Üímajor}`.
- **`scripts/e1-contract.sh`** does the same live, plus asserts the two E1
  extensions *beyond* the oracle and their intentional divergence:
  `$type-change ‚Üí major` (the oracle is blind to exported tokens ‚Üí `patch`) and
  `migration ‚Üí migration` (the oracle, keying on `variantId`, reads it as
  `major`).

Parity holds on the matrix because those deltas are clones of one baseline, so
`variantId`-keying (oracle) and `path`-keying (E1) cluster identically; they
diverge only exactly where E1 is *designed* to be better (uuid churn, migration,
`$type`).

## The authored fixture (why it had to be synthesized)

No shipped template combines components + variants + applied tokens + a
`tokens.json` (PLAN3 risk 5: `penpot-design-system` is 100% legacy / 0 tokens;
`tokens-starter-kit` is 0 components). `scripts/e1-fixture.py` synthesizes a
schema-shaped normalized `.penpot` tree with all three legs and BOTH variant
models, then derives the delta matrix by clone-and-edit ‚Äî the spike's
`make_deltas` approach, so the python oracle can classify the same trees the
Rust extractor reads. The generator is the source of truth (like N1's torture
fixture); the gate regenerates it fresh each run, which is what makes the gate
run-twice idempotent.

## Exit criteria (PLAN3 E1 ‚Äî verbatim, all met)

`scripts/e1-contract.sh` (`just contract`), run-twice idempotent:

- **over an authored fixture pair extracts `{(set name/path, exposedProperties[],
  tokensUsed[])}`** ‚Äî block (a): both variant models + the library surface
  asserted field-by-field. ‚úì
- **proves `extract(A) == extract(A')` where A' is A after settle_to_fixpoint
  (uuid churns; keyed by name/path never the remapped `variantId`)** ‚Äî block (b):
  uuid churn (import-as-new simulation), contract body byte-identical, `diff` =
  PATCH; the oracle's `variantId`-keyed MAJOR noise asserted as the contrast. ‚úì
- **classifies a curated delta matrix (impl-only `$value`‚Üípatch; added‚Üíminor;
  removed/renamed/`$type`-changed‚Üímajor) matching the oracle exactly** ‚Äî block
  (c): four-case oracle parity + the `$type` extension. ‚úì
- **special-cases legacy‚Üífirst-class migration so it does NOT read as a spurious
  minor** ‚Äî block (d): classified `migration`, asserted `!= minor`. ‚úì
- **delete-index-db + reindex rebuilds identical contracts from disk (invariant
  1)** ‚Äî block (e): E1 persists **no** index (extraction is a pure function of
  disk; contracts are computed on demand, not stored), so this is satisfied by
  construction ‚Äî re-extraction is byte-identical, and there is no derived DB to
  wipe. Persisted contract rows arrive in E4 (`DocKind::Package`), where the
  same `sync_core` semantic-hash gate that N1 uses will carry the invariant. ‚úì

## Caveats that were harder than the spike implied

- **The spike's own key was the bug.** The spike report frames caveat 2 as a
  future concern and its `extract_contract.py` cheerfully keys first-class sets
  on `variantId`. Productionizing it meant *diverging* from the oracle's keying
  while *keeping* verdict parity on the shared matrix ‚Äî so E1 groups by
  `variantId` (correct clustering) but emits identity as `path` (correct
  diffing). The gate now asserts the oracle is WRONG on churn, which is a
  stronger, more honest claim than "we match the oracle".
- **Migration reads as major, not minor, under the oracle.** The spike report
  predicted a spurious *minor* (assuming path-keying on both sides); the actual
  `variantId`-keyed oracle reads the migration as *major* (old path-set removed
  + new id-set added). Either way the naive verdict is wrong; E1's `migration`
  label is the only correct reading, and it required the path-keying from caveat
  2 to even be *detectable* (both versions must land on the same key to compare).
- **`$type` sensitivity is outside the oracle entirely.** `extract_contract.py`
  emits no library-level exported tokens, so "`$type`-changed ‚Üí major" has no
  oracle counterpart. E1 models exported tokens as `(path, $type)` so `$value`
  edits stay `patch` while `$type` flips bump `major`; this is asserted against
  the Rust classifier alone, and the gate documents the oracle's blindness
  rather than pretending parity.
- **`tokensUsed` is a set-union at set granularity** (spike caveat 4, carried
  forward): repointing one shape's token to another the set still uses reads as
  an addition, not a loss. Correct semantics ("does the set still depend on token
  X?"), documented so E-later per-variant needs know to extract per-component.

## Follow-up fix (differ keying, post-verification)

Independent verification found a latent misclassification: `diff_contracts` keyed its
before/after lookup by the set `path` alone, while `extract_contracts` keys identity by
`(set_kind, set)`. In a partially-migrated file where a first-class variant set and a legacy
path-convention set share the same `path`, the two contracts collided in the differ and one
shadowed the other ‚Äî a **MAJOR contract loss could read as PATCH** (silent, worst-direction).
Fixed by keying the diff by `(set_kind, set)` (matching the extractor) with the
legacy‚Üífirst-class migration handled as an explicit same-path pairing pass, so coexisting sets
never mask each other and migrations are still detected. Regression test
`diff_same_path_across_kinds_does_not_mask_a_major`; oracle parity + all E1 exit criteria still
green. This matters because E2 hangs a `contractHash` off this differ.
